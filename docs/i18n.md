# i18n

Use i18n to keep UI text localizable, consistent, and reviewable.

This guide focuses on app developer workflow.

---

## Why i18n in UX3

i18n support helps you:

- remove hardcoded UI strings from templates
- support multiple locales safely
- validate key usage during build
- keep terminology consistent across views

---

## Recommended Structure

Store locale resources under your `ux/i18n` folder.

Common patterns:

- single-file locale (for small apps)
- split-by-domain locale files (for larger apps)

Use stable key namespaces such as:

- `common.*`
- `actions.*`
- `home.*`
- `errors.*`

---

## Template Usage

Bind translation keys in templates instead of literal strings.

```html
<h1>{{i18n.home.title}}</h1>
<button ux-event="RETRY">{{i18n.actions.retry}}</button>
<p>{{i18n.errors.network}}</p>
```

For consistency:

- keep keys semantic
- avoid deep, unclear nesting
- reuse existing keys where meaning is shared

---

## Key Design Guidelines

1. Prefer intent-based keys over page-specific duplication.
2. Keep text values concise and UI-focused.
3. Use placeholders for dynamic values.
4. Avoid putting HTML markup inside translation strings.

Example placeholder style:

```json
{
	"validation": {
		"maxLength": "Must be {max} characters or less"
	}
}
```

---

## i18n and FSMs

FSM state templates should reference i18n for:

- state titles
- action labels
- error/retry messages
- empty/loading hints

This keeps all user-facing text centralized while FSM behavior remains declarative.

---

## i18n and Content

Use this split for maintainability:

- i18n: UI chrome, actions, status, validation, navigation
- content: authored long-form markdown pages

If a string is part of app interaction, keep it in i18n.

---

## Validation and Build

Run build/validation regularly:

```bash
npm run build
```

Catch issues early:

- missing keys
- stale/unused keys
- invalid locale resource shape

---

## Testing i18n

Add tests for:

- rendering key strings in core views
- fallback behavior for missing keys
- key coverage for critical user flows

For production apps, include at least one smoke test per supported locale.
